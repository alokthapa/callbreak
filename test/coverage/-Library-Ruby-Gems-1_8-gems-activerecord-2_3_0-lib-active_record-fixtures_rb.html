  24 # Fixtures are a way of organizing data that you want to test against; in short, sample data. They come in 3 flavors:
  25 #
  26 #   1.  YAML fixtures
  27 #   2.  CSV fixtures
  28 #   3.  Single-file fixtures
  29 #
  30 # = YAML fixtures
  31 #
  32 # This type of fixture is in YAML format and the preferred default. YAML is a file format which describes data structures
  33 # in a non-verbose, human-readable format. It ships with Ruby 1.8.1+.
  34 #
  35 # Unlike single-file fixtures, YAML fixtures are stored in a single file per model, which are placed in the directory appointed
  36 # by <tt>ActiveSupport::TestCase.fixture_path=(path)</tt> (this is automatically configured for Rails, so you can just
  37 # put your files in <tt><your-rails-app>/test/fixtures/</tt>). The fixture file ends with the <tt>.yml</tt> file extension (Rails example:
  38 # <tt><your-rails-app>/test/fixtures/web_sites.yml</tt>). The format of a YAML fixture file looks like this:
  39 #
  40 #   rubyonrails:
  41 #     id: 1
  42 #     name: Ruby on Rails
  43 #     url: http://www.rubyonrails.org
  44 #
  45 #   google:
  46 #     id: 2
  47 #     name: Google
  48 #     url: http://www.google.com
  49 #
  50 # This YAML fixture file includes two fixtures.  Each YAML fixture (ie. record) is given a name and is followed by an
  51 # indented list of key/value pairs in the "key: value" format.  Records are separated by a blank line for your viewing
  52 # pleasure.
  53 #
  54 # Note that YAML fixtures are unordered. If you want ordered fixtures, use the omap YAML type.  See http://yaml.org/type/omap.html
  55 # for the specification.  You will need ordered fixtures when you have foreign key constraints on keys in the same table.
  56 # This is commonly needed for tree structures.  Example:
  57 #
  58 #    --- !omap
  59 #    - parent:
  60 #        id:         1
  61 #        parent_id:  NULL
  62 #        title:      Parent
  63 #    - child:
  64 #        id:         2
  65 #        parent_id:  1
  66 #        title:      Child
  67 #
  68 # = CSV fixtures
  69 #
  70 # Fixtures can also be kept in the Comma Separated Value format. Akin to YAML fixtures, CSV fixtures are stored
  71 # in a single file, but instead end with the <tt>.csv</tt> file extension
  72 # (Rails example: <tt><your-rails-app>/test/fixtures/web_sites.csv</tt>).
  73 #
  74 # The format of this type of fixture file is much more compact than the others, but also a little harder to read by us
  75 # humans.  The first line of the CSV file is a comma-separated list of field names.  The rest of the file is then comprised
  76 # of the actual data (1 per line).  Here's an example:
  77 #
  78 #   id, name, url
  79 #   1, Ruby On Rails, http://www.rubyonrails.org
  80 #   2, Google, http://www.google.com
  81 #
  82 # Should you have a piece of data with a comma character in it, you can place double quotes around that value.  If you
  83 # need to use a double quote character, you must escape it with another double quote.
  84 #
  85 # Another unique attribute of the CSV fixture is that it has *no* fixture name like the other two formats.  Instead, the
  86 # fixture names are automatically generated by deriving the class name of the fixture file and adding an incrementing
  87 # number to the end.  In our example, the 1st fixture would be called "web_site_1" and the 2nd one would be called
  88 # "web_site_2".
  89 #
  90 # Most databases and spreadsheets support exporting to CSV format, so this is a great format for you to choose if you
  91 # have existing data somewhere already.
  92 #
  93 # = Single-file fixtures
  94 #
  95 # This type of fixture was the original format for Active Record that has since been deprecated in favor of the YAML and CSV formats.
  96 # Fixtures for this format are created by placing text files in a sub-directory (with the name of the model) to the directory
  97 # appointed by <tt>ActiveSupport::TestCase.fixture_path=(path)</tt> (this is automatically configured for Rails, so you can just
  98 # put your files in <tt><your-rails-app>/test/fixtures/<your-model-name>/</tt> --
  99 # like <tt><your-rails-app>/test/fixtures/web_sites/</tt> for the WebSite model).
 100 #
 101 # Each text file placed in this directory represents a "record".  Usually these types of fixtures are named without
 102 # extensions, but if you are on a Windows machine, you might consider adding <tt>.txt</tt> as the extension.  Here's what the
 103 # above example might look like:
 104 #
 105 #   web_sites/google
 106 #   web_sites/yahoo.txt
 107 #   web_sites/ruby-on-rails
 108 #
 109 # The file format of a standard fixture is simple.  Each line is a property (or column in db speak) and has the syntax
 110 # of "name => value".  Here's an example of the ruby-on-rails fixture above:
 111 #
 112 #   id => 1
 113 #   name => Ruby on Rails
 114 #   url => http://www.rubyonrails.org
 115 #
 116 # = Using Fixtures
 117 #
 118 # Since fixtures are a testing construct, we use them in our unit and functional tests.  There are two ways to use the
 119 # fixtures, but first let's take a look at a sample unit test:
 120 #
 121 #   require 'web_site'
 122 #
 123 #   class WebSiteTest < ActiveSupport::TestCase
 124 #     def test_web_site_count
 125 #       assert_equal 2, WebSite.count
 126 #     end
 127 #   end
 128 #
 129 # As it stands, unless we pre-load the web_site table in our database with two records, this test will fail.  Here's the
 130 # easiest way to add fixtures to the database:
 131 #
 132 #   ...
 133 #   class WebSiteTest < ActiveSupport::TestCase
 134 #     fixtures :web_sites # add more by separating the symbols with commas
 135 #   ...
 136 #
 137 # By adding a "fixtures" method to the test case and passing it a list of symbols (only one is shown here though), we trigger
 138 # the testing environment to automatically load the appropriate fixtures into the database before each test.
 139 # To ensure consistent data, the environment deletes the fixtures before running the load.
 140 #
 141 # In addition to being available in the database, the fixtures are also loaded into a hash stored in an instance variable
 142 # of the test case.  It is named after the symbol... so, in our example, there would be a hash available called
 143 # <tt>@web_sites</tt>.  This is where the "fixture name" comes into play.
 144 #
 145 # On top of that, each record is automatically "found" (using <tt>Model.find(id)</tt>) and placed in the instance variable of its name.
 146 # So for the YAML fixtures, we'd get <tt>@rubyonrails</tt> and <tt>@google</tt>, which could be interrogated using regular Active Record semantics:
 147 #
 148 #   # test if the object created from the fixture data has the same attributes as the data itself
 149 #   def test_find
 150 #     assert_equal @web_sites["rubyonrails"]["name"], @rubyonrails.name
 151 #   end
 152 #
 153 # As seen above, the data hash created from the YAML fixtures would have <tt>@web_sites["rubyonrails"]["url"]</tt> return
 154 # "http://www.rubyonrails.org" and <tt>@web_sites["google"]["name"]</tt> would return "Google". The same fixtures, but loaded
 155 # from a CSV fixture file, would be accessible via <tt>@web_sites["web_site_1"]["name"] == "Ruby on Rails"</tt> and have the individual
 156 # fixtures available as instance variables <tt>@web_site_1</tt> and <tt>@web_site_2</tt>.
 157 #
 158 # If you do not wish to use instantiated fixtures (usually for performance reasons) there are two options.
 159 #
 160 #   - to completely disable instantiated fixtures:
 161 #       self.use_instantiated_fixtures = false
 162 #
 163 #   - to keep the fixture instance (@web_sites) available, but do not automatically 'find' each instance:
 164 #       self.use_instantiated_fixtures = :no_instances
 165 #
 166 # Even if auto-instantiated fixtures are disabled, you can still access them
 167 # by name via special dynamic methods. Each method has the same name as the
 168 # model, and accepts the name of the fixture to instantiate:
 169 #
 170 #   fixtures :web_sites
 171 #
 172 #   def test_find
 173 #     assert_equal "Ruby on Rails", web_sites(:rubyonrails).name
 174 #   end
 175 #
 176 # = Dynamic fixtures with ERb
 177 #
 178 # Some times you don't care about the content of the fixtures as much as you care about the volume. In these cases, you can
 179 # mix ERb in with your YAML or CSV fixtures to create a bunch of fixtures for load testing, like:
 180 #
 181 #   <% for i in 1..1000 %>
 182 #   fix_<%= i %>:
 183 #     id: <%= i %>
 184 #     name: guy_<%= 1 %>
 185 #   <% end %>
 186 #
 187 # This will create 1000 very simple YAML fixtures.
 188 #
 189 # Using ERb, you can also inject dynamic values into your fixtures with inserts like <tt><%= Date.today.strftime("%Y-%m-%d") %></tt>.
 190 # This is however a feature to be used with some caution. The point of fixtures are that they're stable units of predictable
 191 # sample data. If you feel that you need to inject dynamic values, then perhaps you should reexamine whether your application
 192 # is properly testable. Hence, dynamic values in fixtures are to be considered a code smell.
 193 #
 194 # = Transactional fixtures
 195 #
 196 # TestCases can use begin+rollback to isolate their changes to the database instead of having to delete+insert for every test case.
 197 # They can also turn off auto-instantiation of fixture data since the feature is costly and often unused.
 198 #
 199 #   class FooTest < ActiveSupport::TestCase
 200 #     self.use_transactional_fixtures = true
 201 #     self.use_instantiated_fixtures = false
 202 #
 203 #     fixtures :foos
 204 #
 205 #     def test_godzilla
 206 #       assert !Foo.find(:all).empty?
 207 #       Foo.destroy_all
 208 #       assert Foo.find(:all).empty?
 209 #     end
 210 #
 211 #     def test_godzilla_aftermath
 212 #       assert !Foo.find(:all).empty?
 213 #     end
 214 #   end
 215 #
 216 # If you preload your test database with all fixture data (probably in the Rakefile task) and use transactional fixtures,
 217 # then you may omit all fixtures declarations in your test cases since all the data's already there and every case rolls back its changes.
 218 #
 219 # In order to use instantiated fixtures with preloaded data, set +self.pre_loaded_fixtures+ to true. This will provide
 220 # access to fixture data for every table that has been loaded through fixtures (depending on the value of +use_instantiated_fixtures+)
 221 #
 222 # When *not* to use transactional fixtures:
 223 #   1. You're testing whether a transaction works correctly. Nested transactions don't commit until all parent transactions commit,
 224 #      particularly, the fixtures transaction which is begun in setup and rolled back in teardown. Thus, you won't be able to verify
 225 #      the results of your transaction until Active Record supports nested transactions or savepoints (in progress).
 226 #   2. Your database does not support transactions. Every Active Record database supports transactions except MySQL MyISAM.
 227 #      Use InnoDB, MaxDB, or NDB instead.
 228 #
 229 # = Advanced YAML Fixtures
 230 #
 231 # YAML fixtures that don't specify an ID get some extra features:
 232 #
 233 # * Stable, autogenerated ID's
 234 # * Label references for associations (belongs_to, has_one, has_many)
 235 # * HABTM associations as inline lists
 236 # * Autofilled timestamp columns
 237 # * Fixture label interpolation
 238 # * Support for YAML defaults
 239 #
 240 # == Stable, autogenerated ID's
 241 #
 242 # Here, have a monkey fixture:
 243 #
 244 #   george:
 245 #     id: 1
 246 #     name: George the Monkey
 247 #
 248 #   reginald:
 249 #     id: 2
 250 #     name: Reginald the Pirate
 251 #
 252 # Each of these fixtures has two unique identifiers: one for the database
 253 # and one for the humans. Why don't we generate the primary key instead?
 254 # Hashing each fixture's label yields a consistent ID:
 255 #
 256 #   george: # generated id: 503576764
 257 #     name: George the Monkey
 258 #
 259 #   reginald: # generated id: 324201669
 260 #     name: Reginald the Pirate
 261 #
 262 # Active Record looks at the fixture's model class, discovers the correct
 263 # primary key, and generates it right before inserting the fixture
 264 # into the database.
 265 #
 266 # The generated ID for a given label is constant, so we can discover
 267 # any fixture's ID without loading anything, as long as we know the label.
 268 #
 269 # == Label references for associations (belongs_to, has_one, has_many)
 270 #
 271 # Specifying foreign keys in fixtures can be very fragile, not to
 272 # mention difficult to read. Since Active Record can figure out the ID of
 273 # any fixture from its label, you can specify FK's by label instead of ID.
 274 #
 275 # === belongs_to
 276 #
 277 # Let's break out some more monkeys and pirates.
 278 #
 279 #   ### in pirates.yml
 280 #
 281 #   reginald:
 282 #     id: 1
 283 #     name: Reginald the Pirate
 284 #     monkey_id: 1
 285 #
 286 #   ### in monkeys.yml
 287 #
 288 #   george:
 289 #     id: 1
 290 #     name: George the Monkey
 291 #     pirate_id: 1
 292 #
 293 # Add a few more monkeys and pirates and break this into multiple files,
 294 # and it gets pretty hard to keep track of what's going on. Let's
 295 # use labels instead of ID's:
 296 #
 297 #   ### in pirates.yml
 298 #
 299 #   reginald:
 300 #     name: Reginald the Pirate
 301 #     monkey: george
 302 #
 303 #   ### in monkeys.yml
 304 #
 305 #   george:
 306 #     name: George the Monkey
 307 #     pirate: reginald
 308 #
 309 # Pow! All is made clear. Active Record reflects on the fixture's model class,
 310 # finds all the +belongs_to+ associations, and allows you to specify
 311 # a target *label* for the *association* (monkey: george) rather than
 312 # a target *id* for the *FK* (<tt>monkey_id: 1</tt>).
 313 #
 314 # ==== Polymorphic belongs_to
 315 #
 316 # Supporting polymorphic relationships is a little bit more complicated, since
 317 # Active Record needs to know what type your association is pointing at. Something
 318 # like this should look familiar:
 319 #
 320 #   ### in fruit.rb
 321 #
 322 #   belongs_to :eater, :polymorphic => true
 323 #
 324 #   ### in fruits.yml
 325 #
 326 #   apple:
 327 #     id: 1
 328 #     name: apple
 329 #     eater_id: 1
 330 #     eater_type: Monkey
 331 #
 332 # Can we do better? You bet!
 333 #
 334 #   apple:
 335 #     eater: george (Monkey)
 336 #
 337 # Just provide the polymorphic target type and Active Record will take care of the rest.
 338 #
 339 # === has_and_belongs_to_many
 340 #
 341 # Time to give our monkey some fruit.
 342 #
 343 #   ### in monkeys.yml
 344 #
 345 #   george:
 346 #     id: 1
 347 #     name: George the Monkey
 348 #     pirate_id: 1
 349 #
 350 #   ### in fruits.yml
 351 #
 352 #   apple:
 353 #     id: 1
 354 #     name: apple
 355 #
 356 #   orange:
 357 #     id: 2
 358 #     name: orange
 359 #
 360 #   grape:
 361 #     id: 3
 362 #     name: grape
 363 #
 364 #   ### in fruits_monkeys.yml
 365 #
 366 #   apple_george:
 367 #     fruit_id: 1
 368 #     monkey_id: 1
 369 #
 370 #   orange_george:
 371 #     fruit_id: 2
 372 #     monkey_id: 1
 373 #
 374 #   grape_george:
 375 #     fruit_id: 3
 376 #     monkey_id: 1
 377 #
 378 # Let's make the HABTM fixture go away.
 379 #
 380 #   ### in monkeys.yml
 381 #
 382 #   george:
 383 #     name: George the Monkey
 384 #     pirate: reginald
 385 #     fruits: apple, orange, grape
 386 #
 387 #   ### in fruits.yml
 388 #
 389 #   apple:
 390 #     name: apple
 391 #
 392 #   orange:
 393 #     name: orange
 394 #
 395 #   grape:
 396 #     name: grape
 397 #
 398 # Zap! No more fruits_monkeys.yml file. We've specified the list of fruits
 399 # on George's fixture, but we could've just as easily specified a list
 400 # of monkeys on each fruit. As with +belongs_to+, Active Record reflects on
 401 # the fixture's model class and discovers the +has_and_belongs_to_many+
 402 # associations.
 403 #
 404 # == Autofilled timestamp columns
 405 #
 406 # If your table/model specifies any of Active Record's
 407 # standard timestamp columns (+created_at+, +created_on+, +updated_at+, +updated_on+),
 408 # they will automatically be set to <tt>Time.now</tt>.
 409 #
 410 # If you've set specific values, they'll be left alone.
 411 #
 412 # == Fixture label interpolation
 413 #
 414 # The label of the current fixture is always available as a column value:
 415 #
 416 #   geeksomnia:
 417 #     name: Geeksomnia's Account
 418 #     subdomain: $LABEL
 419 #
 420 # Also, sometimes (like when porting older join table fixtures) you'll need
 421 # to be able to get ahold of the identifier for a given label. ERB
 422 # to the rescue:
 423 #
 424 #   george_reginald:
 425 #     monkey_id: <%= Fixtures.identify(:reginald) %>
 426 #     pirate_id: <%= Fixtures.identify(:george) %>
 427 #
 428 # == Support for YAML defaults
 429 #
 430 # You probably already know how to use YAML to set and reuse defaults in
 431 # your <tt>database.yml</tt> file. You can use the same technique in your fixtures:
 432 #
 433 #   DEFAULTS: &DEFAULTS
 434 #     created_on: <%= 3.weeks.ago.to_s(:db) %>
 435 #
 436 #   first:
 437 #     name: Smurf
 438 #     <<: *DEFAULTS
 439 #
 440 #   second:
 441 #     name: Fraggle
 442 #     <<: *DEFAULTS
 443 #
 444 # Any fixture labeled "DEFAULTS" is safely ignored.

 502   def self.create_fixtures(fixtures_directory, table_names, class_names = {})
 503     table_names = [table_names].flatten.map { |n| n.to_s }
 504     connection  = block_given? ? yield : ActiveRecord::Base.connection
 505 
 506     table_names_to_fetch = table_names.reject { |table_name| fixture_is_cached?(connection, table_name) }
 507 
 508     unless table_names_to_fetch.empty?
 509       ActiveRecord::Base.silence do
 510         connection.disable_referential_integrity do
 511           fixtures_map = {}
 512 
 513           fixtures = table_names_to_fetch.map do |table_name|
 514             fixtures_map[table_name] = Fixtures.new(connection, File.split(table_name.to_s).last, class_names[table_name.to_sym], File.join(fixtures_directory, table_name.to_s))
 515           end
 516 
 517           all_loaded_fixtures.update(fixtures_map)
 518 
 519           connection.transaction(:requires_new => true) do
 520             fixtures.reverse.each { |fixture| fixture.delete_existing_fixtures }
 521             fixtures.each { |fixture| fixture.insert_fixtures }
 522 
 523             # Cap primary key sequences to max(pk).
 524             if connection.respond_to?(:reset_pk_sequence!)
 525               table_names.each do |table_name|
 526                 connection.reset_pk_sequence!(table_name)
 527               end
 528             end
 529           end
 530 
 531           cache_fixtures(connection, fixtures_map)
 532         end
 533       end
 534     end
 535     cached_fixtures(connection, table_names)
 536   end
